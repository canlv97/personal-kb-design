# 把 Karpathy 的 Wiki 拆了重装：一份有活人感的知识库搭建手记

> 我们不是在做 Karpathy 教程。我们是从自己的需求出发，发现他给的地图不够用，然后一个坑一个坑填过来的。

---

## 不是"学 Karpathy"，是"修 Karpathy"

网上很多 LLM Wiki 的实现——比如 [llmwiki](https://github.com/lucasastorian/llmwiki)——是把 Karpathy 的模式原样工程化。这件事有价值，但不是我们想做的。

我们做的事不一样：我和我的 AI agent 从头搭建了一个**个人知识库**，过程中发现 Karpathy 的原始方案有好几处跟真实需求对不上。不是他错了——是他的场景（单域 AI 研究）和我们的场景（一个人混乱而真实的知识生活）之间有裂缝。这篇文章记录的是这些裂缝在哪，以及我们怎么填的。

我们觉得这比"又一个 Karpathy 实现"更有分享价值。因为填裂缝的过程本身是一环扣一环的——每一步都是上一步逼出来的。如果你也在搭自己的知识库，大概率会踩到同样的坑。

---

## 第一环：个人知识库天然是多域的

Karpathy 的场景是一个人深入研究一个领域。但真实的人不是这样的。

我的 Obsidian 里有交易笔记、嵌入式开发文档、半导体物理、ASMR 自媒体策划、健康管理、装机配置、考研计划、日常日记。这些东西互相之间毫无关系——它们唯一的共同点是：都属于我。

这不是我的问题，这是**个人知识库的本质**。一个人的知识生活就是跨领域的。甚至企业内部的知识库也一样——华为的知识库如果只放通信协议不放芯片设计、不放美学规范、不放营销案例，那跟没有差不多。

所以第一件事就定了：**这个知识库不能是单域的。** 它必须同时装下交易、AI、嵌入式、光电、自媒体、健康……而且这些域之间天然会产生碰撞。嵌入式中断优先级的仲裁逻辑，和交易中多头信号同时触发时的取舍——底层是不是同一个结构？你不知道，但你想知道。

这就引出了第二环。

---

## 第二环：多域碰撞需要灵感，灵感需要容器

如果你想在六个域之间找方法论迁移，你不能靠关键词搜索。搜"中断"不会命中"止损"，搜"优先级"不会命中"风控"——它们是字面上完全不同的东西。结构同构是你看出来的，不是搜出来的。

但"看出来"这件事不是坐在桌前就能发生的。它往往是一闪念——洗澡的时候、走路的时候、读到某个完全不相关的东西的时候。

所以多域知识库天然需要一个**灵感容器**。一个你可以快速记下"诶，这个是不是可以用到那边？"而不用考虑格式、引用、完整性的地方。

Karpathy 的模式里没有这个东西。他的 wiki 假设你已经有确定的知识了。但多域交叉恰恰依赖于那些尚未确定、尚未验证的"半成品想法"。

卢曼的卡片笔记法把这种状态叫 #Fleeting 闪念笔记。于是我们把 fleeting 缝进了 Karpathy 的体系里。

---

## 第三环：fleeting 不能是独立文件夹

如果你给 fleeting 一个独立文件夹，它就会变成**灵感坟场**。丢进去的东西永远不会再看——因为它跟你的知识体系在物理上是隔离的。

我们的做法：fleeting 不是文件夹，是 `status` 字段的一个值。fleeting 笔记放在 `wiki/literature/` 下面——跟文献笔记、草稿笔记在同一个目录里。它们的区别只在前端那一行：

```yaml
status: fleeting   # 刚记下来的，还没消化
status: draft      # 补充了来源，有了引用链
status: done       # 随时可以升格为永久笔记
```

好处是：lint 的时候 agent 能自动标出所有 30 天没动过的 fleeting。要么发展，要么删。不会默默腐烂。

---

## 第四环：多域意味着索引必须拆

Karpathy 的 index.md 是一个全局列表——所有概念混排。27 个页面的时候还好。六个域、100 个页面的时候，agent 回答一个交易问题要先读完所有域的索引。

我们的做法：每个域一个独立索引。`wiki/交易-index.md` 只有 15 行，200 token。agent 问交易只读交易索引，不问光电绝不碰光电索引。

跨域搜索另走一条路：用 `search_files(output_mode=files_only)` 先定位——这个模式只返回文件名，token 接近零——然后再精读命中的几篇。

## 缝完之后的完整结构

```
我的库/
├── SCHEMA.md                   ← agent 行为规范（活的，会随着纠错而更新）
├── raw/                        ← 原始素材（平铺，分类靠 frontmatter）
├── wiki/
│   ├── index.md                ← 总目录（指向各域索引）
│   ├── log.md                  ← 操作日志
│   ├── 交易-index.md           ← 域索引
│   ├── literature/             ← 文献笔记（status: fleeting → draft → done）
│   ├── concepts/               ← 永久笔记
│   ├── entities/               ← 实体（人、公司、标的、模型）
│   └── comparisons/            ← 对比分析
├── projects/                   ← 项目笔记（临时，有死期）
└── 今花今拾/                   ← 日记（不动）
```

每个 wiki 页面的 frontmatter 跑两套信息：

```yaml
---
# agent 用的检索键
domain: ['交易']
type: concept
status: done

# 人用的卡片标签（保留原有习惯）
tags:
  - literature文献/技巧卡
  - 交易
---
```

## 几个用了才知道的设计决策

**域用 tag，素材类型用文件夹。** 素材类型（论文/文章/转录）是互斥的——一篇东西不可能同时是论文又是文章，文件夹天然适合。域是多对多的——一篇"PID 在量化交易中的应用"同时属于嵌入式和金融，tag 天然适合。两个维度各用各的。

**raw/ 平铺。** 不按域分子文件夹。跨域素材不需要纠结放哪——它就在 raw/ 下面，frontmatter 打两个 domain tag。

**域索引目前靠 agent 手动维护。** 新建页面时自动更新，但如果它写错了没有自动修复。将来可以写个脚本扫描 frontmatter 自动生成——这是最值得优化的地方。

---

## 诚实说：还不完美

- **域索引维护有遗忘风险。** agent 忘了更新索引的话，那个域就丢了一页。目前没有自动校验。
- **raw/ 平铺在手机上不好翻。** 文件管理器里 50 个文件名排一列确实痛苦。Obsidian 里用 Dataview 按 domain 分组可以绕过去。
- **fleeting → permanent 的升级依赖人的判断。** 标准是"综合 2+ 来源"，但实际使用中经常拖。
- **跨域 wikilink 断了不容易发现。** lint 能检测，但得定期跑。

---

## 如果你也想搭一个

1. **别上来就设计完美架构。** 丢 30 篇素材进去，用两周。你会自然知道哪些分类是真的，哪些是你想象的。
2. **SCHEMA.md 是活的。** agent 做错一次你就往里面加一条规则。它会越来越聪明。
3. **域索引是目前最值的投入。** 检索效率越早搞定越好——东西多了再改成本高。
4. **把 lint 设成定时任务。** 断链、过期 fleeting、缺 frontmatter——这些不会自己消失。
